вҸә Here are 3 approaches вҖ” all static HTML, no framework, fits the workspace
  stack:

  ---
  Option A вҖ” Instagram-as-Catalog вӯҗ Recommended

  The Instagram feed is the product catalog. An embed widget (e.g. SnapWidget
  free tier or a lightweight JS solution) pulls @bellacaramella's posts
  directly. Every new post she makes automatically appears on the site вҖ” zero
  maintenance, zero code. Static HTML pages for Home, Nosotras, Contacto.
  WhatsApp CTAs everywhere.

  Pros: Truly zero-effort for the founder. The catalog is always current. No
  JSON to manage.
  Cons: Layout is grid-only, no per-product detail pages. Relies on Instagram
  embed staying live.

  ---
  Option B вҖ” JSON Catalog + Instagram Feed

  Products live in a products.json file (name, category, image URL,
  description). The site reads it on load and renders category-filtered product
  cards. A separate Instagram section shows the latest feed. When a new product
  arrives, you add one object to the JSON.

  Pros: Full control over product layout, categories, detail pages, WhatsApp
  pre-filled messages per product.
  Cons: Still requires you (Jonathan) to update the file вҖ” not truly
  founder-self-service.

  ---
  Option C вҖ” Google Sheet as CMS

  Products live in a published Google Sheet. The site fetches it as a CSV on
  page load and renders the catalog. Founder adds a row вҶ’ site updates
  automatically.

  Pros: Genuinely zero-code for the founder (she fills in a spreadsheet).
  Cons: Extra setup, Google dependency, slightly slower load. Overkill for a
  boutique with ~20вҖ“30 products.

  ---
  My recommendation: Option A for launch.

  The founder already runs the business through Instagram. Her content workflow
  doesn't change at all вҖ” she posts, the site updates. If she later wants proper
   per-product pages (with pre-filled WhatsApp messages per item), we can layer
  Option B on top as Phase 2.

  Does this work for you?
