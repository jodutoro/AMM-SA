05 — Agency FAQ

Common Pain Points

The questions every agency asks in the first week. Answered so you don't have to figure them out the hard way.

Q 01
Do I need to know how to code?
No. Warp + natural language is enough. Type what you want, Claude handles the tool calls. Use /slash commands for repeatable workflows — no scripting required.
Q 02
Why is my token budget draining fast?
Claude reads files in loops. Add .claudeignore immediately (exclude node_modules/, .git/, logs, binaries). Run /compact at 65–70% context. Use /clear at 85% for a fresh start.
Q 03
A tool call failed — what do I do?
Read the error message. Parameter Validation Errors contain the correct schema inside them. Extract the expected parameters, then retry once with the correct call. Don't retry blindly.
Q 04
How do I keep CLAUDE.md effective?
Keep it under 150 lines. One fact per line — no long prose. Prune every 2 weeks. Client IDs belong in brand vault, not here. Dead rules waste context every session.
Q 05
How do I share results with clients or my team?
Use /send-slack, /send-discord, /send-email, or /send-circle. All four integrations are pre-wired — just set your webhook URLs in .env once during setup.
Q 06
Can I run workflows for multiple clients?
Yes. Every /run-* command asks which business to target first. Run /my-account at session start to get your full client list, then run any workflow against the right one.
Q 07
What if Claude makes a decision it shouldn't?
It will ask before any destructive or irreversible action — publishing content, launching ads, sending comms. Never skip confirmations. The cost of a bad publish is higher than the cost of one extra "yes."
Q 08
How do I discover what tools are available?
Call any SearchAtlas tool with op: "help" and params: {} to get its full operation list. Then call a specific op with empty params to get that operation's schema. Always do this before your first real call.

07 — Non-Negotiables

Six Golden Rules

Break any of these and you'll waste time debugging entirely avoidable problems. Read them once, remember them always.

01
Schema Discovery First
Before calling ANY tool for the first time, call it with {} to discover the real schema. Documentation may be outdated — the actual API response is authoritative. Do this once, get it right every time after.
02
Read Error Messages
Parameter Validation Errors contain the correct schema inside them. Don't retry blindly — read the error, extract the correct parameters, retry once. Internal Server Errors are backend issues, not yours.
03
Poll Async Tasks
Many operations return a task ID, not results. Poll with get_otto_task_status until status = SUCCESS. Use otto_wait between polls. Never assume an operation completed because it returned.
04
Watch for Tool Name Collisions
Short names can resolve to the wrong underlying tool. If a tool behaves unexpectedly, try the full prefixed name — e.g., otto_project_management instead of project_management.
05
Never Hardcode IDs
Project IDs, location IDs, business IDs — always discover via API. Run /my-account first. Hardcoded IDs break silently when clients are recreated or transferred.
06
Never Expose Secrets
API keys live in .env only. Never print them. Never include them in Slack, Discord, or email output. Add .env* to both .claudeignore and .gitignore immediately after setup.
